    GPU Simulator Implementation Status                                                                                               
                                                                                                                                      
    Overview                                                                                                                          

    We are building a cycle-accurate C++ simulator for a custom GPU architecture (single SM, 32-thread SIMT warps, RV32IM +
    VDOT8/TLOOKUP custom instructions). The simulator uses a decoupled functional/timing model where the functional model executes
    instructions immediately and the timing model consumes trace events to model cycle-accurate pipeline behavior.

    The full plan is at /Users/colinbaird/Projects/GPU/.claude/plans/valiant-popping-ember.md.

    Known Issues / Things To Watch For

    1. The writeback arbiter fill buffer can only hold one entry — If both a coalescing cache hit and an MSHR fill arrive in the same
    cycle, only one can be submitted. The timing model currently prioritizes MSHR fills. This may need a proper queue.
    2. Serialized load coalescing writeback — When loads serialize to 32 individual cache requests, the current implementation sends
    the full warp result array for each individual request. This is functionally correct (the functional model already computed all
    results) but the timing of when the warp's scoreboard clears may not perfectly match hardware behavior. The first cache hit would
    produce a writeback, but subsequent hits for the same warp would be redundant.
    3. Branch resolution happens at dispatch (operand collector output) rather than at execute stage output. The plan says "resolved
    in execute stage" but the current implementation resolves when the operand collector outputs the instruction (since the TraceEvent
     already has the branch outcome). This is slightly earlier than the spec intends — the branch penalty may be 1-2 cycles less than
    the real hardware. Should be moved to after the ALU unit produces its result.
    4. ECALL handling — Currently handled as a special case in dispatch_to_unit(). ECALL marks the warp inactive immediately. The spec
     says ECALL marks the warp inactive, which is correct, but we should verify the timing (the warp should complete the current
    instruction before becoming inactive, which it does since ECALL reaches the dispatch stage).
    5. Stores don't set scoreboard pending — This is correct per the plan (stores have no rd), but verify that the timing model
    handles the store completing without any writeback entry correctly.

    Key Architectural Decisions Made

    1. Synchronous pull model: Functional model called at issue time (not decode time) to avoid wrong-path execution corruption
    2. Forward-order evaluation with double-buffered scoreboard: Ensures 1-cycle minimum RAW gap
    3. All execution units share the same accept/has_result/consume_result interface via the ExecutionUnit abstract base class
    4. LD/ST results bypass execution unit result buffers — they flow through the cache/MSHR fill path to the writeback arbiter's fill
     buffer
    5. Branch resolution at dispatch output (noted above as a potential timing inaccuracy)
