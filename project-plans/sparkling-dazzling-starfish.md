# Close the Reg-family Migration — Strict Hardware-Faithful Wrapping

## Context

The `current_mut()`-elimination refactor ([`project-plans/goofy-humming-dream.md`](goofy-humming-dream.md), Phases 0–8, currently shipped through HEAD `6d5be5a`) wrapped most timing-model state in `Reg<T>` / `RegFifo<T>` / `PulseReg<T>` / `Wire<T>` and deleted `Reg<T>::current_mut()`. It deliberately deferred a set of structures whose existing semantics could not be wrapped without redesigning a cross-stage or cross-class handshake. The plan documented these as `// deliberate-non-register` exceptions with rationales but punted the wrapping work.

This plan closes those gaps. Every field that **models hardware state** must end up as a Reg-family wrapper (`Reg<T>`, `RegFifo<T>`, `PulseReg<T>`, or `Wire<T>`), and where existing behavior depends on a non-hardware-faithful same-cycle plain-field read/write, the handshake itself is rewritten with new `Wire<T>` signals. **No new primitives are needed** — the existing four are sufficient. The deferred FIFOs that previous attempts mis-diagnosed as needing a new primitive ("asymmetric commit gating", "in-evaluate producer→consumer") in fact map cleanly onto `RegFifo<T>` once the clock-enable domains are properly separated; the discipline rules are documented under *FIFO clock-enable discipline* below. Sim-instrumentation (`// sim-instrumentation`-annotated counters: free-running cycle counters, stat accumulators, peak observers) stays plain — those fields do not model clocked hardware. The lint then enforces the strict taxonomy so the carve-out set is closed and machine-verified.

Performance regressions from per-cycle container copies are accepted; correctness and hardware fidelity are the priority (this matches the user's stated stance in goofy-humming-dream.md).

The migration runs per [`/resources/refactor_workflow.md`](/resources/refactor_workflow.md): one phase = one commit, regression-as-contract, consolidation reviews every ~3 phases.

### Implementer latitude — handshake fidelity is the goal, not byte-identity

**Read this before starting any phase.** The point of this refactor is hardware fidelity. If a phase's existing behavior turns out to depend on a non-hardware-faithful read or write — a same-cycle visibility shortcut, a mid-evaluate write to committed state, an immediate `push_back`/`pop_front` on a committed deque, a "stage's gated commit also gates a FIFO that shouldn't be frozen", a CDC FIFO collapsed into one evaluate body — **you are empowered to redesign the handshake.** Concretely, the following moves are pre-authorized for any phase, no orchestrator check-in required:

- **Add new `Wire<T>` signals** to mediate cross-stage backward or forward signals.
- **Add new sub-stages** to `TimingModel::tick()` (this is exactly what Phase 5 does to DRAMSim3) — class doesn't need to inherit `PipelineStage`; a lightweight class with `evaluate()`/`commit()` and a sweep slot is enough.
- **Reorder the per-tick sweep** if the new ordering produces a more hardware-faithful read of cross-stage state. Document the new order and the rationale.
- **Change cross-class virtual interfaces** (e.g., `ExternalMemoryInterface::get_response()` becomes `current_front()` + `stage_pop()`). Test fixtures get updated to match — the Phase-4 PulseReg precedent of interposing `commit()` calls in test helpers is the canonical pattern.
- **Move ownership of shared state to `TimingModel`** when a structure is touched by more than one stage. The `WarpState[]` ownership pattern is the precedent.
- **Update test fixture call sequences** to drive the new commit boundaries. Do not preserve a test's old call sequence at the cost of leaving a non-faithful handshake in place. If a test was implicitly relying on a same-cycle shortcut, the test's expectations move with the fix.

**Cycle deltas are accepted if they are revealed pre-existing fidelity bugs.** "Byte-identical expected" in the phase table is the *prediction* — it means the analysis says the wrap should be observationally equivalent. It is *not* a contract you must preserve at the cost of fidelity. If a delta emerges:
1. First confirm the wrap itself is semantically faithful (no copy-paste mistake routing a committed read through a staged accessor or vice versa).
2. If still divergent, localize via `--trace` / Perfetto (see `resources/trace_and_perf_counters.md`); identify the field and mechanism.
3. If the divergence comes from the old code relying on a non-faithful read/write, **accept the new cycle counts as correct.** Do not add combinational gating, dual-writes, or sweep-order hacks to reconstruct the old observation. Document the delta and mechanism in the *Per-phase findings* entry; the new numbers become the baseline for subsequent phases.
4. Only treat a delta as a migration mistake if step (3) finds no fidelity justification.

This is the same delta-triage policy as `goofy-humming-dream.md` and `refactor_workflow.md` — restated here so each subagent reads it without chasing references.

**When to pause and ask the orchestrator** (these are the only cases that require check-in):
- You'd need to introduce a new primitive (a new variant of `Reg`/`RegFifo`/`PulseReg`/`Wire`). The plan asserts the existing four are sufficient; a counter-example needs orchestrator sign-off before landing.
- You'd need to expand scope beyond the phase's inventory row (touch fields the row doesn't cover).
- You'd need to unwrap or de-discipline a field the plan says to wrap (e.g., reclassify something as `// sim-instrumentation` that the inventory marks as hardware state).
- The phase is genuinely impossible without an architectural-spec change.

**Default: proceed.** Structural ambiguities that aren't on the above list — pick the cleaner option, document the choice in the commit message, move on. The consolidation reviews catch drift across phases; no need to stall on judgment calls inside a phase.

### What "hardware state" means in this plan

A field models hardware state if it is a clocked register, a hardware FIFO, a combinational signal between two synchronous pipeline stages, or a cross-cycle architectural-visible quantity (PCs, per-warp enable, scheduler bitmaps, MSHR slots, cache tags, etc.). A field is `// sim-instrumentation` if it is an observability-only counter that has no corresponding hardware register in the architectural spec — DRAMSim3's `fabric_cycle_` / `dram_ticks_` / `phase_` / `max_*_queue_`, `LdStUnit::fifo_total_pushes_`, `WarpScheduler::ldst_issued_total_`, `TimingModel::cycle_` (the simulator's meta-observation of its own progress; not a hardware register because the GPU architectural spec doesn't say "I have a 64-bit cycle counter at offset X"). Borderline judgment calls are flagged in the per-phase brief.

### FIFO clock-enable discipline — the key insight for Phases 3–6

The prior migration's "deliberate-non-register" carve-outs for `addr_gen_fifo_`, the `FixedLatencyMemory` queues, and DRAMSim3's internal queues are based on a mis-diagnosis. A synchronous FIFO in RTL has exactly one clock and two independent enable signals (`wr_en`, `rd_en`). It is never "frozen as a unit"; it always clocks, and the enable signals control whether push and pop happen. When today's code asserts that "the FIFO's commit must be gated with one stage but not another," what's actually being modeled is:

1. **The producer's pipeline registers** have a clock-enable wired to `!writeback_stall` (or whatever the freeze condition is). When asserted, those registers hold.
2. **The FIFO's `wr_en`** is a combinational AND of the producer's "I'm ready to push" signal with `!writeback_stall`. When stalled, `wr_en` falls to 0 combinationally — not because the FIFO is frozen, but because its enable is masked.
3. **The FIFO's `rd_en`** is driven by the consumer, independent of the producer's clock-enable.
4. **The FIFO itself** is normally clocked. It samples `wr_en` and `rd_en` on every clock edge.

Today's code conflates (1) and (2) into a single "gate the producer's whole `commit_all()`" trick, which over-freezes the FIFO and breaks the consumer's pop. To work around the over-freeze, the consumer pops directly (immediate `pop_front()`) — exactly the same anti-pattern class as the `current_mut()` write-to-committed-state that the prior refactor eliminated.

The hardware-faithful encoding uses the existing `RegFifo<T>` primitive plus a discipline:

- **Cross-stage FIFOs are not owned by any pipeline stage.** They are declared as direct members of `TimingModel` (the orchestrator), passed by reference / back-pointer to the producer and consumer stages that touch them. `TimingModel::tick()` runs a dedicated ungated `commit()` pass over these cross-stage FIFOs, distinct from the per-stage gated `commit_all()` calls. This makes the cross-stage role visible at the declaration site (the FIFO is a peer of the stages, not a member of one) and removes the "is this stage's gating correct for this FIFO?" question entirely — by construction, a cross-stage FIFO never sits inside a stage's gated commit. Borrowed pattern: `WarpState[]` is already owned by `TimingModel` and passed by reference to fetch / decode / scheduler.
- **The producer conditions `stage_push()` on the backward stall Wire in its `evaluate()`.** That conditional is the literal simulator translation of the RTL `wr_en && !stall` AND-gate. The writeback-stall `Wire` is already a backward combinational signal driven before any consumer's evaluate reads it in the back-to-front sweep.
- **The consumer calls `stage_pop()` unconditionally** (modulo its own pop decision). No new Wire needed; the pop intent goes directly on the `RegFifo`.

Trace a stall cycle:
- LdStUnit `evaluate()`: sees `!wb_arbiter_->next_writeback_stall() == false`, skips `stage_push`. The producer's pipeline registers (`busy_`, `cycles_remaining_`, `pending_entry_`) stay frozen because LdStUnit's own gated `commit_all()` doesn't fire.
- CoalescingUnit `evaluate()`: stages a pop on the FIFO normally.
- LdStUnit `commit()`: gated, early-returns — pipeline state holds.
- FIFO `commit()` (ungated): applies the pop, no push was staged → FIFO shrinks by one. Pipeline of producer holds.
- Next cycle: producer's `pending_entry_` still valid, `cycles_remaining_` still 0. If stall releases, `evaluate()` stages the push; FIFO grows.

This is byte-identical to today's behavior and is a literal translation of the RTL. The earlier Phase-3-of-prior-migration cycle deltas almost certainly came from enrolling the FIFO in the producer's *gated* `RegisteredStage`, which over-freezes the FIFO on stalls and blocks the consumer's pop. Enrolling it ungated avoids the over-freeze.

**The same discipline applies to cross-class memory queues** (`FixedLatencyMemory::responses_/write_acks_`, DRAMSim3 internal queues): the FIFO has one set of clock-enable signals, owned by whichever side is permissive. When cache pops from `responses_` during its own evaluate before mem_if's evaluate runs, the cache is reading the FIFO's `current()` head — that's last-cycle's committed completion, which is what hardware would see anyway (one-cycle response latency). `stage_pop()` from cache during its evaluate, `stage_push()` from mem_if at the tail of mem_if's evaluate, FIFO commit ungated — matches RTL exactly.

### Three shapes — `Reg<container>` vs `RegFifo<T>` vs CDC `RegFifo<T>`

The word "FIFO" overloads three different hardware shapes. The right primitive depends on **who pushes, who pops, and on which clock**:

| Shape | When to use | Storage primitive | Examples |
|-------|-------------|-------------------|----------|
| **Intra-stage staged container** | Single owner mutates the container during its own `evaluate()`. Same-evaluate visibility required — a later mutation must see an earlier one. No cross-stage handshake. Models a flat array of register slots clocked together, with intra-stage logic deciding which slots latch. | `Reg<std::deque<T>>` / `Reg<std::vector<T>>` / `Reg<std::array<T,N>>` / `Reg<std::unordered_map<>>` with `next_mut()` mutations | `MultiplyUnit::pipeline_` (multi-stage iterative pipeline with embedded per-entry latency counters), cache tag array, MSHR file, `LoadGatherBufferFile::buffers_` |
| **Same-clock cross-stage FIFO** | Producer and consumer are different stages on the **same clock**. Producer-push and consumer-pop intents stage independently and apply atomically at commit. No same-evaluate visibility of push to pop — that's the one-clock-cycle latency a hardware FIFO presents. | `RegFifo<T>` declared on `TimingModel`, committed in the dedicated ungated cross-stage commit pass | `addr_gen_fifo_` (LdStUnit → CoalescingUnit), `FixedLatencyMemory::responses_/write_acks_` (mem_if → cache) |
| **CDC FIFO (cross-clock-domain)** | Producer and consumer are on **different clocks** (fabric vs DRAM clock). Push happens on one clock edge, pop on a different clock's edge. In real RTL the boundary uses Gray-coded pointers through a 2–3 flop synchronizer; at minimum one full source-clock cycle of latency before the destination side sees a push. The DRAM side may pop multiple times per fabric cycle if `ticks_per_fabric_ > 1`. | `RegFifo<T>` declared on `TimingModel`, **with each clock domain represented as a separate stage** in `TimingModel::tick()`. RegFifo's pop count is `uint32_t pops_` (extended from the existing `bool pop_`) to support multiple pops per fabric cycle. | `DRAMSim3::request_fifo_` (fabric submit → DRAM-clock issue), `DRAMSim3::responses_` / `write_acks_` (DRAM completion → fabric cache consumer) |

The intra-stage container is **not a hardware FIFO** in any meaningful sense — it's a Reg whose value happens to be a deque. `MultiplyUnit::pipeline_` is the canonical example: per-entry latency counters, in-place mutation of every entry, head-pop when ready. `RegFifo<T>` would actively get in the way (it can't decrement per-entry counters).

The CDC FIFO is structurally a real hardware FIFO **with one extra requirement**: the producer and consumer live in different stages with different sweep positions in `TimingModel::tick()`. The fabric-clock side does its evaluate, then the DRAM-clock side does its evaluate, then the ungated cross-stage commit pass latches both push intents and pop intents into the FIFO. The fabric→DRAM-clock latency that the current `submit_*` then drain-loop pattern skips is the one fabric cycle the commit-deferred RegFifo naturally provides.

### Discipline as a rule

**Same-clock**: when a FIFO has independent clock-enable domains on its read and write ports on the same clock, its commit is owned by the more-permissive (ungated) domain via a TimingModel-owned `RegFifo<T>`, and the more-gated domain's write/read is conditioned in `evaluate()` via the existing backward stall `Wire<T>`.

**Cross-clock**: when a FIFO bridges two clock domains, each domain runs as its own stage in `TimingModel::tick()`, the FIFO is a TimingModel-owned `RegFifo<T>` with the producer staging push and the consumer staging pop (possibly N times per fabric cycle on the DRAM side), and commit is ungated at the fabric-clock cycle boundary.

The lint enforces this indirectly — every FIFO must be `RegFifo<T>` enrolled in *some* `register_state()`; no immediate `push_back`/`pop_front` on committed deques.

## New primitives to introduce — none

The existing four primitives (`Reg<T>`, `RegFifo<T>`, `PulseReg<T>`, `Wire<T>`) are sufficient for every phase in this plan.

The cross-stage / cross-class handshakes (`addr_gen_fifo_` "asymmetric gating", `mem_if` cross-class consumption, DRAMSim3 in-evaluate producer→consumer) were originally diagnosed as needing a new primitive. The RTL analysis in *FIFO clock-enable discipline* above corrects that diagnosis: each of these is a `RegFifo<T>` whose commit lives in an ungated `RegisteredStage`, with the producer-side push (or consumer-side pop) conditioned in `evaluate()` via the existing backward stall Wire. No `AsymmetricRegFifo<T>` or `IntraEvaluatePipe<T>` variant is justified — those would be different surface APIs for the same underlying hardware semantics that `RegFifo<T>` already models.

If during Phase 3, 4, or 5 a subagent discovers a genuine semantic gap (e.g., a FIFO with truly independent producer/consumer clock domains, which the project does not yet have), that's an orchestrator decision before introducing a variant. The default assumption is that no new primitive is needed and every phase converts using the existing four.

## Cross-stage handshake redesigns — design notes

These are the substantive design choices each subagent will execute:

- **Cross-stage FIFO ownership** (applies to Phases 3, 4, 5): all FIFOs touched by more than one pipeline stage are declared as direct members of `TimingModel` (not of any stage), passed by reference / back-pointer to the stages that touch them. `TimingModel::tick()` runs a dedicated ungated `commit()` pass over these cross-stage FIFOs, distinct from per-stage `commit_all()` calls. Pattern borrows from how `WarpState[]` is already owned by `TimingModel` and shared by reference across fetch / decode / scheduler. This pattern is established in Phase 3 and reused in Phases 4–5.
- **`addr_gen_fifo_`**: today CoalescingUnit calls `ldst.pop_front()` directly inside its `commit()` — a `current_mut()`-class immediate write to the committed deque. The new design wraps the FIFO as `RegFifo<AddrGenFIFOEntry>` declared on `TimingModel`, committed in the dedicated cross-stage commit pass. LdStUnit `evaluate()` calls `stage_push()` only when `!wb_arbiter_->next_writeback_stall()` — the RTL `wr_en && !stall` AND-gate. CoalescingUnit `evaluate()` calls `stage_pop()` on its existing pop decision. Both intents apply at the FIFO's ungated commit. On a stalled cycle: producer's pipeline registers freeze (LdStUnit gated commit), producer skips `stage_push`, consumer's `stage_pop` still applies → FIFO drains, producer holds. On the resumed cycle: producer's `pending_entry_` is still valid (its commit was gated), `stage_push` fires, FIFO grows. Matches RTL exactly.
- **`FixedLatencyMemory` cross-class queues**: the cache calls `mem_if.get_response()` during `cache.evaluate()` before `mem_if.evaluate()` runs in the sweep. In hardware, the cache reads the FIFO's `current()` head — last cycle's committed completion, presented with the natural one-cycle FIFO latency. The redesign declares `responses_` / `write_acks_` as `RegFifo<MemoryResponse>` on `TimingModel` (cross-stage ownership pattern), committed in the dedicated cross-stage pass. Cache's `evaluate()` calls `stage_pop()`; `mem_if`'s `evaluate()` calls `stage_push()` at the tail of its latency-decrement loop. `in_flight_` is internal to mem_if and stays as `Reg<std::deque<>>` enrolled in mem_if's own `RegisteredStage`. No sweep reordering required.
- **DRAMSim3 CDC FIFOs (Phase 5)**: today `submit_*()` pushes to `request_fifo_` at the top of `evaluate()` and the fabric loop's `while (phase_ >= 1.0)` drain pops from the same `request_fifo_` later in the same `evaluate()` — fabric-clock push and DRAM-clock pop collapsed into one body, hiding the one-fabric-cycle CDC traversal latency a real hardware FIFO presents. The redesign splits `DRAMSim3Memory` into a **fabric-clock stage** (owns `submit_*`, calls `stage_push()`) and a **DRAM-clock stage** (owns the phase loop, `mem_->ClockTick()`, `dram_ticks_`, and per-DRAM-tick `stage_pop()`). The two stages are registered separately in `TimingModel::tick()`. `request_fifo_` is a `RegFifo<PendingChunk>` owned by `TimingModel` (cross-stage ownership pattern), committed in the dedicated ungated cross-stage commit pass. The DRAM stage reads `request_fifo_.current()[pops_staged()]` (peek-after-staged-pops) and accumulates pop intents through the `pops_` counter (Phase 5a). `responses_` / `write_acks_` share the Phase-4-introduced TimingModel-owned `RegFifo<MemoryResponse>` slots — DRAM stage stages pushes from DRAMSim3 completion callbacks; cache stages pops. Tests bypassing `tick()` interpose `commit()` per Phase-4 precedent. Cycle deltas on memory-bound workloads are **expected and accepted** as revealed CDC-latency fidelity bugs; see *Expected cycle deltas — Phase 5 only* above.
- **WarpState deactivation_request Wire**: similar to the existing `ALUUnit::next_redirect_` / `FetchStage::apply_redirect` Wire pattern. Ecall-completion drives a per-warp bit during the execute phase; fetch reads `active_.current() && !deactivation_request_.value()[w]`. The wire resets at the top of every tick (top-of-tick reset matches the `next_redirect_` convention). On the next cycle, the staged-false from `active_.set_next(false)` is now committed, so the wire mask is no longer needed.

Scheduler reads of `current_fifo_total_pushes()` happen at scheduler.evaluate (later in the same sweep) but pick up the prior-tick committed value — byte-identical with pre-migration commit-time-increment timing.
- **Design — accessor delegation.** LdStUnit retains `current_fifo_empty()`, `current_fifo_front()`, `current_fifo_size()`, `current_fifo_entries()` as delegating accessors over the back-pointed FIFO. Many call sites depend on these (scheduler, snapshot, `pipeline_drained`, tests); option (a) chosen for minimal churn. `LdStUnit::pop_front()` is removed — its sole caller (CoalescingUnit) now uses `addr_gen_fifo_->stage_pop()` directly.
- **Sweep order verification.** Coalescing's evaluate runs BEFORE LdStUnit's evaluate (back-to-front: coalescing -> alu/mul/div/tlookup/ldst). The `wb_arbiter_->next_writeback_stall()` Wire is driven first in the sweep, so LdStUnit's evaluate reads it correctly. Coalescing's `stage_pop` and LdStUnit's `stage_push` co-exist in the same cycle; both apply atomically at `commit_cross_stage_fifos()` (pop-then-push). LdStUnit's evaluate reads `current_size()` (start-of-cycle committed) and does NOT credit the consumer's same-cycle stage_pop — the documented one-cycle bubble at FIFO-full + same-tick pop, mirroring the pre-migration `addr_gen_fifo_.size() < fifo_depth_` check.
- **Panic flush.** The cross-stage `addr_gen_fifo_` is intentionally NOT cleared in the `pending_panic_flush_` block — pre-migration `LdStUnit::reset()` was never invoked from the panic-flush cascade either (only scheduler / opcoll / gather_file / wb_arbiter `flush()`s). The panic controller's step-2 drain polls `execution_units_drained()` (which checks `ldst_->current_fifo_empty()`) until quiescent, so in-flight FIFO entries naturally drain through the panic-branch ticks. The cross-stage commit pass runs in BOTH branches, so any pop/push staged during panic ticks still applies.
